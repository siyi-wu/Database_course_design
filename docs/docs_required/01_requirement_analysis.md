# 实验室设备管理系统需求分析

## 1. 编写目的

本文档用于说明实验室设备管理系统的业务背景、用户角色、功能需求、数据需求和约束规则，是后续 ER 建模、数据库设计、程序实现和测试验收的依据。课程要求中的“系统需求分析文档”和“应用内容描述清楚”主要由本文档与 `docs/analysis/02_er_model.md` 共同支撑。

## 2. 业务背景

学校实验室通常拥有显微设备、测量仪器、计算设备、网络设备、安全设备和电源设备等资产。传统人工登记方式容易出现设备状态更新不及时、借用记录缺失、重复借出、维护设备被误借、统计困难等问题。

本系统面向实验室设备台账维护与借还管理场景，使用 MySQL 建立统一数据库，通过表约束、外键、触发器、存储过程、函数、视图和索引保证数据完整性，并提供 React + Express 编写的界面用于演示基本业务操作。

## 3. 系统目标

- 建立设备类别、实验室房间、用户、设备台账和借用记录的集中数据库。
- 支持设备、用户、房间和类别的基本增删改查。
- 支持设备借出、归还、未归还记录查询和状态自动维护。
- 防止维修、报废或已借出的设备被重复借出。
- 提供统计查询、视图查询、索引性能分析、事务并发控制演示、备份恢复和安全授权材料。
- 满足课程设计关于 5 张以上关联表、3 个触发器、2 个带参过程或函数、2 个游标过程、2 个二级索引和规范化过程说明的要求。

## 4. 用户角色

| 角色 | 主要职责 | 典型操作 |
| --- | --- | --- |
| 学生 | 借用实验课程需要的设备 | 查询可用设备、提交借用、归还设备 |
| 教师 | 借用教学或科研设备 | 查询设备状态、查看借还记录、借用设备 |
| 实验室管理员 | 管理设备和借还流程 | 维护台账、处理借还、同步状态、调拨设备 |
| 系统维护人员 | 维护数据库与运行环境 | 初始化数据库、备份恢复、查看索引和数据库对象 |

## 5. 功能需求

### 5.1 基础资料管理

- 设备类别管理：维护类别名称和类别说明，类别名称不能重复。
- 实验室房间管理：维护房间名称、位置和管理员，房间名称不能重复。
- 用户管理：维护借用人姓名、角色和联系方式，支持学生、教师和工作人员三类角色。
- 设备台账管理：维护设备名称、类别、房间、状态、价格和购置日期。

### 5.2 借还业务管理

- 借用登记：选择设备、用户、借用日期和借用天数，系统生成计划归还日期。
- 归还登记：根据借用记录编号登记实际归还日期。
- 状态联动：借用成功后设备自动变为 `Borrowed`；归还后若没有其他未归还记录，设备自动恢复为 `Available`。
- 异常拦截：维修中、已报废、已存在未归还记录的设备不能再次借出。
- 未归还统计：可按用户统计当前未归还设备数量。

### 5.3 查询与统计

- 按设备名称、编号、类别、房间和状态筛选设备。
- 按用户姓名、联系方式或编号查询用户。
- 按设备名称、用户姓名、记录编号和归还状态查询借用流水。
- 使用视图展示设备详情和借用详情，减少应用层重复拼接多表查询。
- 在总览页面展示设备总数、可用数量、借出数量、维修数量、报废数量、借用记录数量、未归还数量和逾期数量。

### 5.4 数据库高级功能

- 使用触发器校验借还日期、设备状态和重复借出规则。
- 使用带参存储过程完成借用和归还操作。
- 使用带参函数统计用户未归还数量。
- 使用游标过程批量同步设备状态、批量调拨可用设备。
- 使用显式事务和 `SELECT ... FOR UPDATE` 演示并发借用控制。
- 使用二级索引优化高频查询。

## 6. 数据需求

系统核心数据实体包括：

- `categories`：设备类别，包括类别编号、名称和说明。
- `labrooms`：实验室房间，包括房间编号、名称、位置和管理员。
- `users`：借用用户，包括用户编号、姓名、角色和联系方式。
- `equipments`：设备台账，包括设备编号、名称、类别、房间、状态、价格和购置日期。
- `borrowrecords`：借用记录，包括记录编号、设备、用户、借用日期、计划归还日期和实际归还日期。

这些实体之间存在明确参照关系：类别和设备、房间和设备、用户和借用记录、设备和借用记录均为一对多关系。

## 7. 业务规则

- 每台设备必须属于一个有效设备类别。
- 每台设备必须放置在一个有效实验室房间。
- 每条借用记录必须对应一个有效用户和一台有效设备。
- 计划归还日期不能早于借用日期。
- 实际归还日期不能早于借用日期。
- 状态为 `Maintenance` 或 `Scrapped` 的设备不能借出。
- 同一设备同一时间只能存在一条 `actual_return_date IS NULL` 的未归还记录。
- 借用成功后设备状态自动改为 `Borrowed`。
- 归还成功后，如果该设备不存在其他未归还记录，设备状态自动改为 `Available`。
- 被外键引用的类别、房间、用户和设备不能被直接删除，以避免历史记录失效。

## 8. 非功能需求

- 完整性：通过主键、唯一约束、外键、枚举字段、触发器和存储过程共同保证数据正确。
- 性能：对类别状态筛选、用户借用记录查询和设备未归还记录检查建立二级索引。
- 安全性：提供最小权限数据库账号授权模板，避免 UI 使用高权限账号。
- 可维护性：SQL 按建表、可编程对象、索引、基准测试、演示和安全分目录管理。
- 可恢复性：提供基于 `mysqldump` 和 `mysql` 的备份恢复脚本。
- 可演示性：提供 UI、测试脚本和测试报告，便于课程验收时展示完整流程。

## 9. 验收边界

本系统重点实现实验室设备台账与借还管理，不包含预约审批、维修工单流转、资产折旧核算、统一身份认证和多级权限审批。这些内容可作为后续扩展方向，不影响本次课程设计核心要求的完成。
